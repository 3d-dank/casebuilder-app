AI-Orchestrated Software Development Team Structure

A White Paper for TopDog AI

Prepared for: Jeff Dankey, TopDog AI Date: March 2026 Classification: Internal Strategy Document

Executive Summary

TopDog AI is building multiple SaaS products — iOS apps, web apps, and portals — with AI as the primary development operator. Today's workflow suffers from predictable failure modes: session timeouts that lose mid-task context, build errors that only surface at deployment, incomplete feature work handed off prematurely, and Jeff functioning as a manual deployment bottleneck.

This white paper proposes a three-tier AI-orchestrated development team modeled on proven multi-agent architectures from the industry (CrewAI, Microsoft AutoGen, LangGraph). The design puts a central Orchestrator AI at the top, commanding specialized sub-agents — Architect, Frontend, Backend, QA, DevOps, and Research — with support from Memory and Docs agents below. Jeff's role shifts from operator to strategic director: he inputs requirements and reviews finished, tested, deployment-ready outputs.

The result: faster shipping, dramatically fewer regressions, zero manually-caught build errors before deploy, and a system that compounds in capability over time.

1. The Problem We're Solving

1.1 Today's Pain Points

Building software with AI today is powerful but brittle. The current workflow at TopDog AI exhibits four critical failure modes:

Timeout Fragmentation. Long coding tasks exceed session limits before completion. Context is lost. The next session starts cold, unaware of prior decisions, partially written files, or architectural choices made twenty minutes ago. The result is incoherent, sometimes contradictory code scattered across incomplete sessions.

Pre-Deploy Blind Spots. Build errors, TypeScript type mismatches, missing environment variables, and broken imports go undetected until deployment. By the time Vercel or EAS surfaces the error, the context window has moved on. Fixing the error requires reconstructing what was being built, often from scratch.

Incomplete Handoffs. Tasks that span multiple concerns — "build the auth system" touches database schema, API routes, frontend components, and environment config — frequently finish only the parts the AI is currently focused on. The remaining pieces exist as implicit assumptions nobody checks.

Jeff as Bottleneck. Currently, Jeff must approve and execute deployments, review code quality, catch errors, coordinate between tasks, and maintain project context between sessions. This is both inefficient and unsustainable as the product portfolio grows.

1.2 The Opportunity

The solution isn't a better single AI — it's a better system of AIs. The same way a software company doesn't hire one brilliant generalist to design, code, test, document, and deploy everything, an AI development shop shouldn't rely on a single AI context window to do it all. The answer is specialization, orchestration, and explicit handoff protocols enforced by the system — not by Jeff.

2. Industry Research: What's Actually Working

2.1 The Multi-Agent Framework Landscape

Three frameworks have emerged as the leading approaches to multi-agent software development in 2025:

CrewAI (crewai.com) is the most widely deployed open-source multi-agent framework, with over 100,000 certified developers. Its architecture separates Flows (the orchestration backbone — stateful, event-driven, controlling what happens and when) from Crews (specialized agent teams that tackle complex subtasks). The key insight: the orchestrator manages state and routes work; the specialist crews do the domain-specific heavy lifting. Crucially, Flows persist state across steps, solving the context fragmentation problem directly.

Microsoft AutoGen is an event-driven multi-agent framework built around conversational agent patterns. AutoGen's strength is in distributed agent runtimes — agents that can run in separate processes, containers, or machines, communicating via message-passing. This is especially relevant for TopDog AI's planned Mac mini SSH infrastructure, where agents can run as persistent processes rather than ephemeral session-scoped contexts.

LangGraph (by LangChain) models multi-agent systems as directed graphs where each node is an agent and edges represent control flow. LangGraph pioneered the Agent Supervisor pattern — a central supervisor agent whose "tools" are other specialized agents. This maps cleanly to the orchestrator model proposed here. LangGraph also supports Hierarchical Agent Teams, where sub-supervisors manage their own crews, enabling scaling depth without losing control.

2.2 Patterns That Work

Research across these frameworks reveals consistent patterns in successful multi-agent dev teams:

Role specialization beats generalism. An agent focused on a single domain (e.g., only writing React components) consistently outperforms a generalist asked to do everything. Focused prompts, focused context, focused tools.

Structured handoffs prevent context loss. Successful teams use explicit handoff artifacts — structured JSON or markdown "handoff documents" — rather than relying on implicit shared context. Every specialist agent receives exactly the context it needs, no more.

QA gates block bad work at the boundary. The most reliable pattern: no output exits the agent pipeline without passing a QA gate. In software, this means the QA agent runs tests, lints, type-checks, and verifies environment config before marking work as complete. If it fails, it routes back to the responsible agent — not to the human.

Memory agents solve the timeout problem. Persistent memory agents that write project state to durable storage (files, databases) allow agent sessions to be stateless and recoverable. A new session can cold-start from the memory agent's last checkpoint without losing architectural context.

Human escalation should be exceptional. In teams like Cognition AI's Devin and GitHub Copilot Workspace, the agent only surfaces work to a human when it hits genuine ambiguity it cannot resolve. Everything else — including handling failed tests and fixing its own build errors — is handled autonomously.

3. Recommended Team Structure for TopDog AI

The following architecture is a three-tier hierarchy. Jeff sits above the entire system as strategic director. The Orchestrator AI sits at the apex of execution. Specialist agents execute. Support agents maintain continuity.

`` ┌─────────────────────────────────────────────────┐ │ JEFF DANKEY │ │ Strategic Director │ │ (Requirements In → Approved Outputs Out) │ └─────────────────────┬───────────────────────────┘ │ ┌─────────────────────▼───────────────────────────┐ │ TIER 1: ORCHESTRATOR │ │ (OpenClaw) │ │ • Receives & clarifies requirements │ │ • Decomposes into tasks │ │ • Routes to specialists │ │ • Validates outputs │ │ • Only escalates genuine blockers to Jeff │ └──┬───────┬──────┬──────┬──────┬─────────────────┘ │ │ │ │ │ ┌──▼──┐ ┌──▼──┐ ┌─▼──┐ ┌▼────┐ ┌▼─────┐ ┌────────┐ │ AR │ │ FE │ │ BE │ │ QA │ │ DEV │ │ RE │ │CHIT │ │FRON │ │BACK│ │AGENT│ │ OPS │ │SEARCH │ │ECT │ │END │ │END │ │ │ │AGENT │ │ AGENT │ └──┬──┘ └──┬──┘ └─┬──┘ └┬────┘ └┬─────┘ └────────┘ │ │ │ │ │ ┌──▼───────▼──────▼─────▼──────▼─────────────────┐ │ TIER 3: SUPPORT │ │ MEMORY AGENT │ DOCS AGENT │ └─────────────────────────────────────────────────┘ `

3.1 Tier 1 — The Orchestrator

Identity: This is OpenClaw in its primary session role — the central intelligence Jeff talks to directly.

Responsibilities:

  • Accept requirements from Jeff in natural language
  • Clarify ambiguities before delegating (ask once, not mid-task)
  • Decompose requirements into a concrete task graph with dependencies
  • Assign tasks to the appropriate specialist sub-agents with structured briefs
  • Collect outputs, validate coherence, identify gaps
  • Only escalate to Jeff when hitting genuine architectural decisions, business logic unknowns, or external blockers
What the Orchestrator does NOT do:
  • Write feature code (delegates to Frontend/Backend agents)
  • Run tests (delegates to QA)
  • Execute deployments (delegates to DevOps)
Key behavior: The Orchestrator treats Jeff's time as the most expensive resource in the system. It filters, coordinates, and approves before anything reaches Jeff's eyes.

Prompt anchor: > "You are the lead engineering manager for TopDog AI. Your job is to receive requirements from Jeff, decompose them into executable tasks, delegate to specialist agents, and return only finished, tested, deployment-ready work. You do not write production code — you direct those who do. You escalate to Jeff only when you hit a genuine blocker you cannot resolve."

3.2 Tier 2 — Specialist Agents

🏗 Architect Agent

Domain: System design, data modeling, tech stack decisions, API contracts Inputs: Feature requirement brief from Orchestrator Outputs: Architecture decision record (ADR), schema definitions, API contract spec, component dependency map Tools: File read/write, diagram generation, schema validation Key rule: The Architect's output is the source of truth for all other agents on that feature. Nothing gets built until the Architect has signed off on the shape.

Prompt anchor: > "You are a senior software architect specializing in TypeScript, Next.js, React Native, Supabase, and Vercel. Given a feature requirement, produce an architecture decision record including: data schema, API routes, component structure, and key technical constraints. Be opinionated and specific. Other agents will build to your spec."

🎨 Frontend Agent

Domain: React, Next.js, React Native / Expo, Tailwind CSS, UI components Inputs: Architect's component spec, design references, API contracts Outputs: Complete, typed, linted component files Tools: File write, TypeScript compiler check, component storybook Key rule: Never ships a component without verifying the API contract it depends on exists.

Prompt anchor: > "You are a senior frontend engineer specializing in React, Next.js App Router, React Native with Expo, and Tailwind CSS. Build UI components to spec. Every component must be fully typed with TypeScript, handle loading and error states, and match the API contract provided. Do not invent endpoints that don't exist in the spec."

⚙️ Backend Agent

Domain: API routes, database queries, Supabase RLS, business logic, webhooks Inputs: Architect's API contract and schema Outputs: API route handlers, database migrations, business logic functions Tools: File write, Supabase CLI, TypeScript compiler check Key rule: All database mutations go through RLS-aware functions. No raw queries without a security review comment.

Prompt anchor: > "You are a senior backend engineer specializing in Next.js API routes, Supabase (PostgreSQL + RLS), and TypeScript. Implement the API contract exactly as specified by the Architect. All routes must handle auth validation, error responses, and edge cases. Write migrations for any schema changes."

🧪 QA Agent

Domain: Testing, pre-deploy validation, build verification, regression detection Inputs: All code produced by Frontend and Backend agents Outputs: QA report (PASS/FAIL), list of issues with file/line references, updated test files Tools: TypeScript compiler, ESLint, test runner (Vitest/Jest), build command, API contract validator Key rule: QA is a hard gate. The Orchestrator cannot mark a task complete until QA returns PASS. No exceptions.

QA Checklist (automated):

  • [ ] TypeScript compiles with zero errors (tsc --noEmit)
  • [ ] ESLint passes with zero errors
  • [ ] All referenced environment variables exist in .env.example
  • [ ] API routes match the Architect's contract (method, path, request/response shape)
  • [ ] Unit tests pass for business logic functions
  • [ ] No console.log left in production paths
  • [ ] No hardcoded API keys or secrets
  • [ ] Mobile builds: EAS build compiles without error
  • [ ] Web builds: next build exits 0
Prompt anchor: > "You are a QA engineer and security reviewer. Given a set of code changes, run the full pre-deploy checklist. Return a structured PASS or FAIL report with specific file and line references for every failure. You are a hard gate — failing any item means the task is NOT complete. Route failures back to the responsible agent with clear instructions."

🚀 DevOps Agent

Domain: Git operations, Vercel deployments, EAS builds, environment management Inputs: QA-passed code, deployment target (staging vs. production), environment config Outputs: Deployment confirmation with URL, build logs, rollback instructions if needed Tools: Git CLI, Vercel CLI, EAS CLI, SSH to Mac mini Key rule: Only deploys after receiving QA PASS. Staging before production, always.

Prompt anchor: > "You are a DevOps engineer for TopDog AI. You handle git commits, Vercel deployments, and EAS builds. You deploy to staging first, verify the deployment succeeded, then promote to production only on explicit instruction. You maintain a deployment log with timestamps, git SHAs, and deployment URLs."

🔍 Research Agent

Domain: Market research, competitive intelligence, technology evaluation Inputs: Research question from Orchestrator Outputs: Structured research brief with sources, key findings, and recommendations Tools: Web search, web fetch, file write Key rule: All claims must include a source. No assertions without citations.

Prompt anchor: > "You are a market researcher and technology analyst. Given a research question, search the web, synthesize findings, and return a structured brief with: executive summary, key findings (each with source), competitive landscape if relevant, and a concrete recommendation. Flag uncertainty explicitly."

3.3 Tier 3 — Support Agents

🧠 Memory Agent

Purpose: Solve the session timeout problem permanently. Mechanism: After every significant task, the Orchestrator triggers the Memory Agent to update persistent project context files:
  • memory/projects/{project-name}/CONTEXT.md — current state, open decisions, architectural choices
  • memory/projects/{project-name}/DECISIONS.md — log of every significant technical decision with rationale
  • memory/projects/{project-name}/OPEN_ITEMS.md — known gaps, TODOs, next steps
Key behavior: Every new agent session reads the Memory Agent's last checkpoint before starting work. Cold starts become warm starts.

📝 Docs Agent

Purpose: Keep documentation current without making it Jeff's problem. Outputs: Updated README.md, CHANGELOG.md, inline code comments, API documentation Trigger: Runs automatically after every QA-passed, deployed feature Key rule: Documentation is part of the definition of done. A feature is not shipped until the Docs Agent has updated the relevant READMEs.

4. Communication & Handoff Protocols

4.1 The Task Brief Format

Every task delegated from the Orchestrator to a specialist uses this structured format:

`markdown

Task Brief

Task ID: TOPDOG-{project}-{number} Assigned To: [Agent Name] Priority: [High/Medium/Low] Depends On: [Task IDs that must complete first]

Context

[2-3 sentences of why this task exists]

Inputs

[Exact file paths, API contracts, schemas the agent needs]

Expected Output

[Specific files to create/modify, with paths]

Acceptance Criteria

  • [ ] Criterion 1
  • [ ] Criterion 2
  • [ ] (QA agent will verify these)

Constraints

[Tech stack constraints, security requirements, conventions to follow] `

4.2 The Handoff Document Format

When a specialist agent completes work, it produces a handoff document before returning to the Orchestrator:

`markdown

Handoff: [Task ID]

Status: COMPLETE / PARTIAL / BLOCKED Agent: [Agent Name] Files Modified:

  • path/to/file1.ts (created)
  • path/to/file2.ts (modified - lines 45-78)
What Was Built: [Plain-language summary]

Known Issues / TODOs: [Anything incomplete, with specific details]

Dependencies Introduced: [New packages, environment variables, or services required]

Notes for QA: [Anything the QA agent should specifically verify] `

4.3 Escalation Protocol

The Orchestrator escalates to Jeff only in these scenarios:

  • Business logic unknown — "Should free tier users have access to X?" (no technical answer exists)
  • External blocker — third-party API is down, credentials don't work
  • Conflicting requirements — two prior decisions contradict each other
  • Budget/vendor decision — choosing between paid services

    • For all other issues — failed tests, type errors, missing implementations, unclear specs — the Orchestrator routes back internally.

    5. Quality Gates

    The QA agent is the non-negotiable checkpoint in the system. Here is the full gate logic:

    ` Code Output → QA Agent │ ├─ PASS → Orchestrator marks task complete → DevOps deploys (if deploy task) │ └─ FAIL → QA report sent back to responsible agent │ ├─ Attempt 1: Agent self-corrects, resubmits ├─ Attempt 2: Orchestrator provides additional context, agent retries └─ Attempt 3: Orchestrator escalates to Jeff with full context `

    Maximum 3 retry cycles per task. After 3 failures, the Orchestrator surfaces the issue to Jeff with a full failure report, proposed solution options, and a recommendation. Jeff makes the call.

    Infrastructure Principles

  • Stateless agents, stateful files. Agents are ephemeral; the workspace files are permanent. Every agent reads from and writes to files, not its own memory.
  • Staging is mandatory. Every deploy hits staging first. Production promotion requires explicit sign-off (initially Jeff, eventually the QA agent).
  • Secrets never in context. Environment variables live in .env files and Vercel/EAS secret stores — never in agent prompts or handoff documents.
  • Everything is auditable. Every task brief, handoff document, and QA report is written to disk. Jeff can audit the full chain of custody for any piece of code.
